 背景
  当前问题在运行态：ACP WebSocket connection 承载了 agent 工作循环。前端发 prompt 后，后端在当前 connection 的 cx.spawn 里跑 agent.reply()，流式事件也
  直接发到这个 connection。WebSocket 一断，connection cleanup 会 abort 任务。结果是同一个持久化 session 可以恢复历史，但运行中的 agent/MCP/request 不能
  跨刷新或跨设备继续。

  goose2web 的持久化数据已经是后端统一的：projects 存在后端 sources，chats/sessions 存在 SQLite，session 里有 project_id。多设备连接同一个 goose serve
  时，project 列表、chat 历史、project-chat 归属基本一致。

  目标是：

  goose serve
    -> 持有 session runtime
      -> 每个 session 一个 Agent/MCP runtime
      -> active prompt 后台运行
      -> 事件进入 session event bus
      -> idle 超过阈值后释放 runtime

  多个网页/设备
    -> attach 同一个 session
    -> 观察同一批事件
    -> 任意设备可发送、取消、批准权限

  计划

  1. 提升 GooseAcpAgent 生命周期。
     现在每个 WebSocket connection 都会创建新的 GooseAcpAgent。改成 AcpServer 在 goose serve 生命周期内持有一个共享 GooseAcpAgent，所有 WebSocket 连接
     复用它。
  2. 引入 serve 级 session runtime registry。
     以 session_id 为 key 保存运行态：

  SessionRuntime {
      agent: AgentHandle,
      active_request: Option<ActiveRequest>,
      event_bus: SessionEventBus,
      pending_permissions: HashMap<PermissionId, PendingPermission>,
      attached_clients: HashSet<ClientId>,
      last_activity_at: Instant,
  }

  Agent/MCP 继续绑定 session，不绑定 WebSocket。

  3. 修改 loadSession 语义。
     如果 session runtime 已存在，loadSession 只 attach/replay，不重建 agent/MCP。
     如果不存在，则从 SQLite 加载 session，创建 runtime，初始化 agent/MCP。
  4. 拆分 on_prompt。
     当前 on_prompt 同步消费 agent.reply() 并直接往 cx 发通知。改成：
     检查 session 是否 busy。
     idle 时注册 active_request。
     用 tokio::spawn 后台运行 agent.reply()。
     后台任务把事件写入 event_bus。
     完成后清理 active_request。
   5. 建立 ACP session event bus。
      建立一个绑定在 SessionRuntime 上的 SessionEventBus。本阶段只做 chat/session 级别，不做全局 bus，也不做 project bus。一个 chat 对应一个持久化
      sessionId，也对应一个 SessionRuntime 和一个 SessionEventBus。

      本阶段目标不是全量替换所有现有 ACP 推送，而是先解决核心运行态问题：

      WebSocket 断开后，active prompt 继续运行
      新 WebSocket attach 同一个 session 后，可以继续收到或补到 chat streaming 事件
      前端可以知道这个 session runtime 是否仍然存活

      本阶段实现范围：

      实现：
         chat 流式事件转发/replay
         session runtime alive 状态
         多个连接订阅同一个 session bus

      暂不实现：
         extension 状态推送
         permission snapshot
         subagent runtime/events
         project/server scope bus
         全量替换所有 cx.send_notification()

      建议结构：

      struct SessionEventBus {
            next_seq: AtomicU64,
            tx: broadcast::Sender<RuntimeEvent>,
            buffer: Mutex<VecDeque<RuntimeEvent>>,
            snapshot: RwLock<SessionRuntimeSnapshot>,
      }

      统一事件 envelope：

      type RuntimeEvent = {
         id: string;
         seq: number;
         sessionId: string;
         kind: string;
         delivery: "replay" | "snapshot" | "ephemeral";
         createdAt: string;
         requestId?: string;
         payload: unknown;
      };

      本阶段实际支持的 kind：

      chat.agent_chunk
      chat.thought_chunk
      chat.tool_call
      chat.tool_call_update
      chat.finish
      chat.error
      session.runtime_state

      delivery 语义：

      chat.*:
         replay
         写入 bounded replay buffer，短线重连时补发。

      session.runtime_state:
         snapshot
         只保留最新状态，重连时通过 snapshot 返回。

      本阶段 snapshot 保持最小：

      type SessionRuntimeSnapshot = {
         sessionId: string;
         alive: boolean;
         activeRequestId?: string;
         lastSeq: number;
      };

      事件发布流程：

      后台 prompt task
         -> 消费 agent.reply()
         -> 将 chat chunk/tool call/tool update/finish/error 发布到 SessionEventBus
         -> delivery=replay 的事件进入 buffer
         -> broadcast 给所有在线 subscribers

      连接 attach/replay 流程：

      client attach/loadSession(sessionId, lastSeq?)
         -> 后端确保 SessionRuntime 存在
         -> 返回 SessionRuntimeSnapshot
         -> 如果 lastSeq 仍在 buffer 范围内，返回 seq > lastSeq 的 replay events
         -> 如果 lastSeq 太旧，返回 replayTooOld=true，前端重新 load persisted chat history
         -> 当前 WebSocket 订阅 live events

      本阶段不是全量替换原有推送。只迁移 prompt streaming 关键路径：

      替换：
         on_prompt 中 chat chunk/tool call/tool update/finish/error 的推送
         session runtime alive 状态

      保留：
         loadSession 历史 replay
         config update
         usage update
         extension/session 其他自定义通知
         非 prompt 路径的 cx.send_notification()

      后续可以逐步把更多事件接入同一个 SessionEventBus：

      extension.status
      extension.list_update
      permission.requested
      permission.resolved
      subagent.status
      subagent.chat_chunk

      但这些不属于本阶段必须实现的内容。
  6. 支持重连和多设备 attach。
     新设备连接后：
     listProjects/listSessions 得到一致的后端数据。
     打开某个 sessionId 后 loadSession/attachSession。
     后端返回历史 replay、当前 active request、pending permission、buffered events。
     前端恢复为 idle/streaming/waiting_permission。
  7. 调整 cancel 语义。
     cancel 以 session_id 为主，任意 attached client 都能取消该 session 当前 active request。
     可带 request_id 防止误取消，但不是权限边界。
  8. 调整权限确认语义。
     pending permission 存在 session runtime。
     所有 attached clients 都能看到。
     任意 client 可 approve/deny。
     后端只接受第一个有效响应，然后广播结果。
  9. 增加 session idle 超时释放。
     新增可配置参数，例如：

  goose:
    session_idle_ttl: 30min

    0 表示永不自动释放，默认建议 30min。

  静默定义：

  没有 active prompt
  没有 pending permission
  没有正在初始化的 agent/MCP
  没有 attached clients
  并且 last_activity_at 超过 idle TTL

  启动 serve 级 sweeper task 周期扫描 runtimes：

  if runtime idle 超过阈值:
    取消/释放 Agent
    关闭 MCP clients/processes

  再次打开该 chat 时，从 SQLite 恢复 session 并重新创建 Agent/MCP，不丢历史。

  10. 前端 session 入口策略。
     同浏览器刷新：优先从 URL 或 localStorage 恢复 active session。
     跨设备：优先 URL 里的 sessionId；没有 URL 时，从后端 active/recent sessions 选择。
     建议新增 ACP 方法：

  _goose/sessions/active

  返回 running/waiting/idle session，帮助新设备发现正在运行的 chat。

  11. 验证。
     测试刷新时 session runtime 不被销毁。
     测试 prompt 运行中断开 WebSocket 后后台任务继续完成。
     测试新 connection attach 后能看到 active/buffered events。
     测试两个客户端 attach 同一 session，B 能取消 A 发起的 active request。
     测试同一 session busy 时第二个 prompt 被拒绝或排队。
     测试 idle 超过 TTL 后 runtime 被释放，SQLite 历史仍可恢复。
